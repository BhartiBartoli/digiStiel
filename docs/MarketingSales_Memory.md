## **The three canonical ICPs**

* **ICP 1 — Ambitious SME owner:** 10–100 employees, owner actively involved, revenue growth a priority, limited internal digital expertise, pragmatic decision process. Attractive: short sales cycle, direct access to decision maker, high reference value.  
* **ICP 2 — Growing service provider:** consultancy, technical services, installers, B2B service firms. Attractive: high value per customer, clear improvement↔revenue link, fast ROI.  
* **ICP 3 — Scale-up in growth/transformation:** 25–250 employees, multiple departments, existing software stack. Attractive: multiple Value Plans, land-and-expand potential, higher contract value.  
* Overarching: **mindset matters more than sector** — outcome-driven, not technology-driven. Customer Zero \= owner of a 5–50 person firm with growth ambition, limited expertise, willingness to change.

  ## **Acquisition model — four-phase evolution (canonical)**

1. **Founder-Led Sales** — networks, referrals; to learn fast, refine ICP, validate use cases.  
2. **Founder-Led \+ Community-Led** — VOKA, entrepreneur networks, sector orgs, events; build trust, thought leadership, references.  
3. **Partner-Led Growth** — consultants, accountants, agencies, implementation partners; scalable lead gen, lower CAC, regional expansion.  
4. **Platform-Led & Content-Led** — Value Discovery Agent, self-assessment, content; requires proven results.

  ## **Brand Architecture (CONFIRMED — permanent GTM principle)**

Cross-chat update from Vision/Platform, adopted as permanent M\&S strategy. A structural GTM decision, **not** a naming or branding exercise. Drives positioning, portfolio, expansion and future acquisitions.

**Five separated layers:** (1) **Corporate Brand** — company, governance, investors, legal, reputation; *not* the primary commercial proposition. (2) **Platform Brand** — the shared AI Value Creation Engine; stable for years, commercial propositions never force changes to it. (3) **Commercial Brands** — the market-facing layer, organised around segments/industries/challenges/value props, *never* around technology/AI/modules. (4) **Products & Services** — Growth Plans, Result Plans, Value Acceleration / Managed / Advisory Services; sit under a Commercial Brand and may change often. (5) **AI Platform** — invisible to most customers; partners/investors recognise the shared engine.

**Six confirmed principles:** (1) Commercial Brands are **GTM engines, not products** — each owns its own ICP, messaging, journey, use cases, pricing, acquisition, partner ecosystem. (2) Technology is never the commercial story; value is. (3) **One platform, multiple market stories** — same engine powers multiple Commercial Brands, each native to its audience. (4) Marketing is organised around markets, not features/modules. (5) Commercial language stays separated from platform language (Growth/Result/Productivity Plan → internally all **Value Plan**). (6) Commercial Brands evolve independently while the platform stays stable — enabling new industries, acquisitions, white-label, international expansion without redesign.

**Long-term GTM shift:** from *one company → one website → one product* toward *one Corporate Brand → one Platform Engine → multiple Commercial Brands → multiple ICPs → multiple market strategies*.

**Investor message:** technology reused, propositions specialised → better capital efficiency, speed to market, gross margins, scalability. (Reinforces the existing "repeatable model across domains" and "increasing value via context" arguments, and mirrors the platform moat: one engine, value in the layers above.)

**Anti-proliferation governance** (apply to every future request for a new brand / product / proposition): (Q1) Is this a genuinely different segment / market / value proposition? If no → extend an existing Commercial Brand. (Q2) Is a new *brand* actually required, or is it just a new service / package / use case / campaign? If the latter → do **not** create a brand. (Q3) Does it improve positioning / GTM / scalability / market clarity? If not → avoid. *Strategy defines architecture; architecture defines brands; brands support value.*

**MVP caution (flagged):** multiple Commercial Brands is a scale-phase ambition (phases 3–5). For MVP and first customers, run **one** Commercial Brand (digiStiel, RevOps beachhead) with the architecture ready underneath — so a later brand needs no redesign. "Safe default \+ deliberate decision." Guard against premature brand splitting during the founder-led, simultaneous BE+NL launch.

  ## **Wallet reporting screens (M\&S design, within frozen Platform frame)**

Post-login wallet \= where the customer sees the value hierarchy. Object model (Brok A/B/C) is frozen; M\&S designs the *experience/copy*, Platform fixes *what is visible*. Frozen frame (not M\&S's to change): customer sees the value hierarchy (goal → plan → operational → indicator) but the Value Indicator only as *value* — `indicatorType`, `supportedBy`, computed aggregation and the substrate are internal. Reporting principles: Understanding → Explanation → Measurement; Narrative Understanding not a dashboard; progressive disclosure (never start at level 5); Home \= "what needs attention today?".

**Klanttaal-mapping (M\&S call — objectmodel-terms fully hidden).** Customer never sees internal terms. Default mapping: Strategic Intent → "waar je naartoe wil"; Strategic Goal → "je doel"; Value Plan → "je plan" (or GroeiPlan/ResultaatPlan…); Operational Goal → "wat je nu doet"; Value Indicator → "je resultaat" (Omzet/Klanten/Retentie…). The mapping is a *default, not a fixed rule* — the platform learns each customer's own words while the internal object stays constant (this is what protects the Reuse moat: a pattern travels cross-tenant even when customer A says "Project" and B says "Traject"). Rests on the frozen principle "digiStiel standardises meaning, not language"; vocabulary is a memory-object, the mapping layer a first-class capability, and Governance explains in the customer's words.

**Home \= "dit vraagt vandaag je aandacht"** — short list, top 2–3 items, rest one click away. Narrative not grid: each item is one sentence a KMO-owner grasps in five seconds. Spectrum shown: an early warning (leading indicator drifting — calm tone, "nog geen groot verschil, daarom zie je het nu"), a plan awaiting customer approval (the Customer Validation Gate as reassurance: "jij beslist, niet wij" — the decisionAuthority default made visible), and a success. **Tone call: calm, not alarmist** — digiStiel is a quiet business partner, not an alarm system; a KMO-owner has enough stress.

**Home Attention — tone layer (M\&S) vs business-priority (Platform/DI).** After a Platform boundary ruling: *which* items deserve attention, the *severity threshold* (when "drifting lightly" becomes "drifting sharply" and rises), and the *ranking/urgency* are business-priority → they belong to Decision Intelligence (the canonical "Attention Candidates", produced from the already-frozen DI signals: Gate \= action, Verdict/Judgement on a leading indicator \= attention, computed-evaluation-ahead \= confirmation). This prevents every channel (web/mobile/AI/PDF/email) building its own priority — same logic as the frozen anti-rule "Projection never performs business interpretation", now extended to the ViewModel. **What stays M\&S (tone/presentation):**

* **Tone markers per signal type** (stub defaults, confirmed): Action/Gate → reassuring, inviting ("er staat iets voor je klaar, jij beslist" — no urgency pressure); Attention/leading-drift → calm, informing ("iets om in de gaten te houden", not "probleem"; on a severe drift DI flags, copy may be more direct but stays factual, never alarmist); Confirmation/success → warm, affirming ("mooi bezig", no exaggeration).  
* **Order within equal priority** (if two candidates share DI-priority, M\&S sets their relative order), copy, card form, the "one sentence in plain language" shape, animation, and the **cap** (Top N, max 2–3 on Home — a presentation choice).  
* **Grounding tone principle (canonical M\&S):** the wallet opens at *what works and the action that's ready*, not at the worry — quiet business partner, not alarm system. Calm but not naïve: a genuinely severe issue may rise, but that severity call is DI's, not a hardcoded M\&S rule. **Input-wish to Platform/DI (not an M\&S rule):** M\&S's preferred prioritisation is action → confirmation → ordinary attention, with a severity threshold that lets an urgent warning rise above confirmation. Formulated as how DI *should* prioritise the Attention Candidates — Platform/DI owns the implementation.

**Plan card title \= computed title derived from the plan's Operational Goals (M\&S derivation rule; supersedes both the earlier "je plan: \<naam\>" fix and the interim goal-context title).** Platform decided (Computed Before Stored, 3rd application after measurableValue and currentness): a Value Plan gets **no stored name field** (Brok A stays frozen); the plan title is *computed* from the Operational Goals the plan carries, in the ViewModel card form (presentation layer, Projection stays "je plan"). Platform bounced the *derivation rule* to M\&S — that's klanttaal.

**The governing principle: the title expresses the value/outcome the plan pursues, NOT the activity.** Operational Goals are "wat je nu doet" — activities. Naming the card after an activity ("Bestaande klanten vinden en gebruiken de webshop") shifts the customer's gaze from value to mechanism — against "waarde staat centraal, nooit het kanaal/de activiteit". So the computed title names the *shared value-direction* the OGs jointly serve, phrased as a short outcome in klanttaal — never the literal OG-name.

Rules: (1) **one OG** → title \= the *outcome* that OG pursues, not the OG-name literally. (2) **multiple OGs** → name the *shared value-direction* they jointly serve (not "first OG", not a mechanical concatenation — both drift to activity/length). (3) **"je plan" stays** — form is **"je plan: \<computed title\>"**; "je plan" \= the standardised meaning, the computed title \= recognisability. Label is augmented, not replaced (klanttaal-mapping intact). This is finer than the interim goal-context title — it also distinguishes plans *within the same Goal* (the limitation goal-context couldn't solve). (4) **identical-OGs edge case** → agree with Platform: if two plans have exactly the same OGs, they ARE the same plan (the plan *is* its OGs) — identical title is a canon-confirmation, not a bug; no separate concept needed.

Demo computed titles (each an *outcome*, not an activity — the canon test): webshop-plan → "je plan: Meer online omzet uit bestaande klanten"; regio-plan → "je plan: Bekend worden in een grotere regio"; projectklanten-plan → "je plan: Eerste projectklanten winnen"; projectproces-plan → "je plan: Een vlotter projectproces".

**One gate-card at a time on Home (M\&S tone input to DI).** Two "wacht op jouw akkoord" cards side by side reads as a to-do list and undermines the "quiet partner, not alarm system" tone. M\&S tone-preference (input-wish to DI, since selection/ranking is DI's): Home shows at most **one** gate/action card at a time — the most relevant; further pending plans wait one click away (or appear once the first is handled). One action at a time respects the calm; two decisions at login feels like pressure. (The cap/Top-N is M\&S's presentation choice; *which* candidates fill it is DI's.) For the demo this is stronger anyway: one gate-card \+ the confirmation \+ the warning spreads the four scenario-types more calmly; the pril new-value- stream plan takes a fourth/click-away slot, fitting its "pril" character.

**Executive Summary of a plan** follows Understanding → Explanation → Measurement, visually labelled in that order: first one sentence on where you stand, then why (with the biggest opportunity marked), then the numbers. Computed `measurableValue` shown as reassurance ("berekend uit je resultaten, geen apart ingevuld getal") — builds trust rather than hiding it. Raw Data / operational detail sit behind buttons (level 4–5 on request, never upfront).

The flow: **Wallet → Home → Executive Summary → Value Story → Operational Detail → Evidence.** Home is just the entrance; this prevents Home from later growing into a dashboard. **Conversation-Centric Workspace (Platform architecture principle, M\&S-confirmed & adopted).** Platform canonised what we implicitly always designed: *the Conversation IS the Workspace* — not a feature, not a chat panel in a corner of a dashboard. The centre of the workspace always holds an *active conversation*; Home is the only exception (pure overview: wat vraagt aandacht / waar sta je / waar wil je naartoe). Opening any object (Goal, Value Plan, Executive Summary, Evidence, Decision, Timeline) does NOT open a detail page — it opens a *conversation about that object* ("hoe bereik ik dit doel sneller?", "waarom loopt dit achter?"). The user never talks to "the application"; he talks to the subject he opened. Three regions: LEFT \= navigation (business-work labels, NOT chat labels — see refinement 1 below); CENTER \= the active conversation about the opened object; RIGHT \= the living context of *that* conversation, which changes when the conversation changes. This is the true expression of "Conversational Value Creation Platform", and it supersedes the "screen/detail-page" mental model — Home \+ Executive Summary as *pages* was the interim build; the direction is conversation-per-object.

**M\&S lane within this (experience/klanttaal — Platform owns the architecture):**

* The RIGHT context region is a **read-only projection**, never a control panel (Presentation Read Model discipline \+ Model B: the customer receives/reads, never operates). It shows only what's *relevant to the current conversation*, kept sparse — max a handful of cards. The moment it becomes a KPI-grid, the wallet is an application again. Klanttaal, not object-names: "inzicht" (not Insights), "onthouden" (not Memory), "jouw beslissing" (not Decisions), "het verhaal" (not Evidence). Show only persistent knowledge that came from the owner himself ("jij wil eerst je bestaande klanten"), not the partner's analysis of him — recognised, not analysed. (Respects "conversations transient, knowledge persistent".)  
* The conversation continues the discovery relationship: same tone as the discovery agent (geen chatbot, één oprechte vraag, mensentaal, builds on the owner's words, verkoopt niks), now with the business objects as shared context. The owner who felt "eindelijk begrepen" in discovery re-enters a space that feels like continuing that conversation, not logging into software.  
* Model B in the conversation: the partner *proposes and explains*, the customer *decides*. The conversation invites ("zal ik het even overlopen?"), never obligates.  
* Navigation language stays conversational ("bekijk het verhaal ernaast", "ga verder"), never "open detail / klik".

## **Customer-facing value/investment dashboard \+ credit economy (V2 — cross-role confirmed)**

Emerged from the demo-broadening (Van Dijck Wonen scale-up). Business Case delivered the economic logic, Vision confirmed the product-model direction; M\&S owns the *experience/tone*. All V2 — activated after proof of proposition 1; the MVP stays "no dashboard, only Cost per Discovery logged". Demo may show the V2 mirror **only with the demo-marking** (projected mature state, not day-1).

**What the customer sees — three quantities, strict hierarchy (Business Case call, M\&S tone):**

1. **Dominant: delivered value** — realised progress toward the customer's own goals, in customer language ("je doel / je plan / je resultaat"). Must dominate the screen.  
2. **Subordinate: what they put in** — never "kosten" or "verbruik". M\&S language call: **"je investering"** or **"ingezet tegoed"**. A calm administrative fact inside the value context, never the headline. **Model B operator-boundary (Vision, canonical):** the credit is shown as *context, never as a control panel*. The customer must never face an operational credit-decision to get value ("do I have enough credit for this action?") — that would make them the operator, which Model B (Managed Value Creation: digiStiel manages, the customer receives) excludes. digiStiel manages consumption on the customer's behalf; the credit is a settlement layer *under* the managed value, not a self-service meter *above* it. If a design forces the customer to operate the credit, it's wrong against Model B and goes back.  
3. **Connecting, only when value/investment is positive: the return** (value/investment ratio). The economic core of "explainable value pricing": the customer sees that what they put in is less than what they get back, without the underlying cent-structure. **Hard boundary:** customer sees *outcome and return, never unit-cost*. No cost-meter (tokens, compute, human-minutes) — a cost-meter is cost-plus logic and invites the customer to negotiate COGS, which value-based pricing avoids. Internal KPIs (ARPA, gross margin, Cost per Discovery) are a *separate* mirror — they measure digiStiel's health and belong on the internal/investor dashboard, never customer-facing (showing them would expose our own margin). Two mirrors, not one.

**Credit/wallet \= a payment mechanism, not a pricing model.** Price stays value-based (access to progress, not AI-usage). The credit (the **Advice Wallet**, already canonical Vision/BC) is how the customer *prefinances* that value-based relationship. What they pay for on top-up \= access to value-creation capacity (expertise, governance, platform effort on their goals), consistent with the discovery-free line (free until digiStiel incurs explicit cost). The credit falls as **value is created, not as compute is consumed** — keep the credit-drain tied to value-units, never usage-units (API-calls), or the positioning breaks. The three cost-potjes (build/run-discovery/ run-delivery) are internal COGS behind the screen; the credit is external (what the customer puts in) and must exceed delivery-COGS (that gap is the margin) — but the customer never sees that relation. Fits maturity-based payment (canonical): Phase 1 \= pay on realised value (no track record yet); Phase 2 \= prepaid wallet becomes logical once trust is built. So wallet \= V2, after proof.

**Product-model (Vision-confirmed, canonical):** the delivered-value core stays dominant; credits/multi-user/cost-visibility are a *serving* V2/V3 layer, NOT a repositioning toward "platform-you-work-in / pay-per-use" (that would push digiStiel back into the SaaS category it explicitly rejects). Cost-visibility presented subordinately within the customer-language discipline keeps the value-first tone intact. Multi-user changes nothing: several people can look at the same delivered value without the product shifting from "value partner" to "workplace". Multi-user touches Identity & Access (Platform, inert, own phase — V3).
